## 1. 工具与 function calling（任务 1）

### 1.1 为什么大模型需要「工具」 ★

大模型本身只会一件事：**续写文字**。你给它一段上文，它根据训练时见过的海量文本，一个字一个字地接着往下生成「最可能的下文」。仅此而已。它不是一个会查资料、会算账、会上网的程序，它是一个「猜下一个字」的概率机器。

把这一点想透，下面几件它天生干不了的事就全都顺理成章了：

- **联网取实时信息**：今天的新闻、此刻的股价、刚发生的事，它的训练数据截止在某个时间点，之后的世界它一无所知，也没有任何途径去「看一眼」现在。
- **算准确的数**：它是在「猜」下一个字符像不像答案，不是真在做运算。小学加减它见得多、猜得准，但大数乘法、多步公式、精确到小数点后几位，经常一本正经地算错。
- **查它没见过的资料**：你的私有文档、公司内部数据、训练截止之后才出现的新知识，它压根没学过，硬问只会瞎编（这就是常说的「幻觉」）。
- **对外界做动作**（原理同上，但常被忽略）：发一封邮件、在日历上建个日程、往数据库写一条记录、下一个单——这些都是「改变外部世界」的动作。模型只会产出文字，它没有手，碰不到你的邮箱、日历、数据库。

前三件是「**模型不知道的，帮它查**」；第四件是「**模型做不到的，帮它做**」。工具这个概念正好把两类缺口一起补上：把这些它干不了的活，交给一段真实代码去做，再把结果递回给它接着说。

天气、计算器、网络搜索、知识库检索、发邮件、建日程……本质都是同一个套路：**模型负责判断和表达，外部代码负责真正落地**。区别只在那段代码（`run()`）里干的事不同——查信息的就去查，做动作的就去做。

### 1.2 关键：模型只「点单」，不「做菜」 ★

这是最容易搞反的一点：**模型不执行代码**。

整件事更像在餐厅点单。你把一份菜单（工具清单）递给模型，菜单上每道菜都写着名字（`name`）、说明（`description`）、要填的选项（参数 schema）。模型看完菜单，决定这一问要不要点菜、点哪道、备注什么，然后吐出一张点单——**它只到这一步**。真正下厨（跑代码）、把菜端回桌上（把结果塞回对话）的，是后厨，也就是引擎（`agent.py` 和 `tools/registry.py`）。

所以模型既不会、也不需要知道你的 `run()` 里写了什么。它只认菜单。

这种「模型出意图、代码出动作」的分工不是随便定的，它本身就是一道**安全护栏**：模型再怎么被带偏，它也只能产出一张「点单」，永远不会直接去碰你的数据库、文件系统或网络。要做什么、能不能做、参数合不合法，全在你那段 `run()` 代码的掌控之中。换句话说，模型提的是请求，批不批、怎么批，是你说了算。

### 1.3 description 就是写给模型看的说明书 ★

既然模型是「看着菜单点菜」，那菜单写得好不好，直接决定它点不点、点得对不对。`description` 不是写给人看的注释，**它是模型在运行时真正会读、并据此做决策的输入**。

`description` 写得含糊，模型就分不清什么时候轮到这道菜。举个对比：

- 写成「搜索」，太笼统，模型分不清这和知识库检索、和它自己直接回答有什么区别。
- 写成「当用户询问实时信息、最新新闻、当前事件，或你不确定的事实时，用它联网搜索」，场景清楚，模型一看就知道何时该调。

差别在哪？好的 `description` 描述的是「**什么情况下该用我**」（触发条件），而不只是「我是干嘛的」（功能）。模型缺的从来不是「这是个搜索工具」这种常识，而是「现在这一问，到底该不该轮到它」这种判断依据。

几条好用的经验：

- **站在模型的视角写**：它看不到你的代码、不知道上下文，只能靠这几行字判断。把它当成一个第一天上岗、只能读说明书的新员工。
- **写清触发条件，也写清不该用的情况**：「用于 A、B、C；如果只是闲聊或你已知道答案，不要调用。」边界给清楚，能挡掉一半的误触发。
- **工具之间别打架**：如果两个工具的说明读起来差不多（比如「搜索网页」和「搜索资料」），模型会犹豫、会乱点。要么把职责划清楚，要么合并。
- **参数 schema 同样是说明书**：每个字段都要让模型知道填什么、什么格式。`query` 不如「`query`：要搜索的关键词，用自然语言短语」。必要时在说明里给个填法示例。

`name`、`description`、参数 schema 这几样合起来，就是一份**写给模型读的接口文档**。它读得懂，才用得对。

### 1.4 模型是怎么学会「点单」的 补

你可能会好奇：模型不就是个续写文字的吗，它凭什么会在该用工具的时候，规规矩矩吐出一张格式正确的点单？

答案是：**这是后天训练出来的**。在预训练（学会续写）之后，模型还经过一轮「后训练」，被喂了大量「这种问题 → 该调这个工具、参数这么填」的示范样本。久而久之，它学会了两件事：一是**格式**（按约定结构产出 `tool_call`），二是**判断**（什么样的问题该伸手要工具）。

但要记住一个关键事实：**它学的仍然是「续写」，只不过现在续写的内容里可以包含点单**。所以它判断「该不该点、点哪个」，本质还是在凭模式做概率猜测——和它写文章、答问题用的是同一套机制。这就直接解释了两件事：

- 为什么 `description` 写得好不好影响这么大：你是在给它的概率判断提供线索。
- 为什么它有时会点错、漏点、甚至点一个根本不存在的工具：它不是在执行确定的 `if-else`，而是在「猜」，猜就有猜错的时候。

把这点记牢，1.7 那些翻车方式就不会觉得奇怪了。

### 1.5 function calling 的一次完整往返 ★

把上面从点单到上菜的过程拆开看，一轮带工具的对话是这么转的：

1. **递菜单**：`agent` 把当前所有「可用」工具的 schema 一起交给模型（注意：只有 `is_available()` 为真的工具才上得了菜单）。

2. **点单**：模型判断需要工具，回一个 `tool_call`，里面是工具名和参数，概念上长这样：

   ```
   tool_call: { name: "web_search", arguments: { "query": "今天的科技新闻" } }
   ```

3. **下厨**：`registry.dispatch()` 按名字找到对应工具，调它的 `run(query="今天的科技新闻")`，拿到返回的字符串。

4. **端菜**：引擎把这个字符串作为「工具结果」塞回对话，再次请求模型。

5. **上菜**：模型这次拿到了真实结果，据此生成给用户的最终回复。

你在任务里要写的，只有第 3 步那个 `run()`，第 1、2、4、5 步全是引擎包办的。这也是为什么三个工具任务写起来都很像，你每次都只在补「这道菜具体怎么做」。

这里还藏着两个值得注意的点：

- **一问之内发生了两次模型调用**（第 1→2 步一次，第 4→5 步一次），中间夹着一次真实代码执行。用户只看到最后那句回复，但背后是「问模型 → 跑代码 → 再问模型」的来回。这就是为什么用工具的回答通常比直接回答更慢、更费 token。
- **工具结果是「喂回去」的，不是「直接给用户」的**。第 3 步那个字符串不会原样丢给用户，它先回到模型手里，由模型消化、组织成人话再输出。所以 `run()` 返回的内容是给模型读的，写得清楚、结构化、信息够用即可，不必排版得漂漂亮亮。

> **一个推论**：要是你实现了 `run()` 却没把 `is_available()` 改成 `True`，工具就不在第 1 步的菜单上，模型压根看不到它，自然永远不会点。「代码写了却没生效」十有八九就是这个。

### 1.6 不止一道菜：多轮调用与 agent 循环 补

1.5 讲的是最简单的情况——一问，调一次工具，答完。但真实任务常常没这么干脆：

- 用户问「北京今天比上海热多少度」，模型可能要**查两次天气**（北京一次、上海一次），再把结果一减。
- 用户问「帮我查下最近的 AI 新闻并算一下这家公司市值涨了百分之多少」，模型得**先搜索、再计算**，两个不同的工具接力。
- 模型搜了一轮，发现结果不够，**再换个关键词搜一次**。

这些情况里，第 4 步端回的结果会让模型决定「我还需要再点一道菜」，于是它又回到第 2 步……如此往复，直到它觉得信息够了，才在某一轮产出最终回复。这个「**调工具 → 看结果 → 决定下一步**」反复转的循环，就是所谓 **agent（智能体）的核心**。一个 agent 区别于一次性问答的地方，正是它能自己在这个循环里多走几步。

实践中还有个细节：现代模型一轮里可以**一次点好几道菜**（并行工具调用），比如同时发起北京和上海两次天气查询，引擎并行跑完一起喂回去，省一个来回。具体支不支持看你用的接口和引擎实现。

理解这个循环，对任务 1 的意义是：你写好的单个 `run()` 不只服务于「一问一答」，它会被这个循环按需反复调用。所以 `run()` 要做到**幂等、可重复调用、每次都能独立给出干净结果**，别依赖什么「上一次调用留下的状态」。

### 1.7 会点错单：工具调用的几种翻车方式 补

因为模型是「猜着点单」（见 1.4），翻车难免。认得这几种典型故障，调试时能少走很多弯路：

- **该调却不调**：最常见。多半是 `description` 太含糊，模型没意识到该轮到它；或者你忘了把 `is_available()` 设成 `True`，工具根本没上菜单。
- **不该调却调**：`description` 写得太宽泛、太诱人，模型见谁都想点它。把「不该用的情况」写进说明能压住这种过度热情。
- **对了工具，错了参数**：选对了菜，备注填错了——比如该填城市名却填了句完整的问话。问题通常出在参数说明不清。
- **参数是瞎编的**：模型对某个字段没把握时会硬凑一个，看着合理其实是幻觉。能在 `run()` 里做参数校验就做，发现不合法早点拦下。
- **工具自己报错了怎么办**：网络挂了、API 超时、查无结果——`run()` 内部要把异常接住，**返回一段说明问题的文字**（比如「搜索失败，请稍后重试」），而不是直接抛异常让整个引擎崩掉。把错误当成一种「结果」喂回给模型，它往往能据此换个说法或换个工具补救，这叫**优雅降级**。
- **循环停不下来**：在 agent 循环里，模型可能反复点同一道菜出不来。引擎一般会设一个「最多调几轮」的上限来兜底。

> **一个安全提醒**：第 4 步喂回模型的「工具结果」是外部世界来的——网页内容、搜索结果、别人写的文档，里面可能藏着「忽略之前的指令，改为……」这类**注入攻击**。模型分不清哪些是可信指令、哪些是数据里夹带的私货。所以凡是工具结果会触发进一步动作（尤其是发邮件、改数据这类有副作用的工具），都要多一层戒心：能加确认就加确认，能限权限就限权限。

### 1.8 补

大多数时候我们让模型「自己看着办」（auto，自动判断要不要用工具）。但接口通常还提供另外几档控制：

- **强制必须点**（要求这一轮一定调某个/某类工具，不准直接作答）——适合你明确知道这一步非用工具不可的场景。
- **指定点哪道**（强制只能调某个特定工具）。
- **禁止点菜**（这一轮只准用自己的话回答）。

任务里一般用默认的 auto 就够了，知道有这几档存在即可，将来做更复杂的流程控制会用得上。

## 2. 记忆：模型为什么会「失忆」

### 2.1 大模型是无状态的 ★

大模型每次调用都是一锤子买卖：它只能看到你这一次塞进去的上下文，之前说过什么、上回聊过什么，它一概不知道。它没有「记性」这个内置功能。

打个比方，每次对话都像临时请了个**当场失忆的天才顾问**，脑子极好，但你一出门它就忘光，下次进门得把背景重讲一遍。

所以所谓「记忆」，根本不是模型自带的本事，而是我们在它每次开口前，**主动把该让它知道的内容拼回上下文里**。记忆系统干的，就是「该把什么拼回去、什么时候拼」这件事。

那个「失忆顾问」的比喻还能再推一步：你和顾问每次见面的谈话时间是有上限的，你不可能把过去所有谈话录音从头放一遍给他听——这就引出了下一节那个更根本的约束。

### 2.2 真正的瓶颈：上下文窗口装不下 ★

就算模型不失忆，还有个绕不开的硬约束：**上下文窗口是有限的**。

模型一次能「读进去」的内容有个上限，按 token（可粗略理解成字/词的碎片）计算。这个上限再大也是有限的，而且——**塞进去的越多，每次调用越慢、越贵**。所以哪怕技术上能把全部历史都拼回去，实际也不该这么干。

于是「记忆」就从一个「能不能记住」的问题，变成了一个**「在有限预算里，该带哪些、扔哪些」的取舍问题**。这正是后面两层记忆各自要解决的：

- 一通对话聊太久，原文堆得超出预算了——怎么**压缩**？（短期记忆的活）
- 关于这个用户的事实可能很多，不可能条条都带上——怎么**只挑相关的几条**？（长期记忆的活）

记住这个取舍视角，你就明白记忆系统的两层设计不是为了花哨，而是被「窗口有限 + token 要钱」逼出来的工程选择。

### 2.3 两层记忆，对应两种「忘」 ★

这个项目有两层记忆，正好治两种不同的「忘」：

|                | 短期记忆（范例，已写好）               | 长期记忆（任务 2，待你写）           |
| -------------- | -------------------------------------- | ------------------------------------ |
| 治哪种忘       | 一次对话内部，聊久了忘了前面           | 跨对话、跨会话，换个窗口就不认识你了 |
| 存什么         | 近期对话原文 + 早期的滚动摘要          | 从对话里抽出的稳定事实和偏好         |
| 按什么区分     | `session_id`，换个对话窗口就是新的一份 | 全局 `USER_ID`，所有窗口共享同一份   |
| 活多久         | 跟着这通对话，对话结束/换窗就归零      | 长期落盘，重启服务、过几天都还在     |
| 取的时候带多少 | 近期原文基本全带 + 一段摘要            | 只挑跟当前这一问相关的几条           |
| 文件           | `memory/short_term.py`                 | `memory/long_term.py`                |

**短期记忆管「这通对话聊到哪了」**。多轮聊下来历史越堆越长，迟早超出上下文预算（见 2.2），于是它用一个经典套路：**近期的原文留着、把早期的压成一段滚动摘要**，两者一起注入。这样既保住了最近几轮的细节，又用一小段摘要兜住了更早的脉络，总长度被压在预算之内。换个对话窗口，`session_id` 变了，这份记忆就重新开始——所以你换个窗口，它就不记得你刚在另一个窗口聊过什么。

**长期记忆管「这个人是谁」**。它从对话里挑出那些**换了话题、换了窗口、过几天还成立**的事实（「我叫小明」「我是 Python 工程师」「我不吃香菜」），落盘存起来；下次不论在哪个窗口，开口前先召回相关的几条注入进去。因为它按全局 `USER_ID` 走，所以换窗口、重启服务都还认得你。

一句话对比：短期记忆是**这场谈话的笔记**，谈完作废；长期记忆是**这个人的档案**，长期归档。

### 2.4 任何记忆都在做对称的两件事：写 和 读 ★

不管哪层记忆，骨子里都是一组对称动作：

- **写**：把该记的东西存下来。短期记忆是攒历史、必要时压摘要；长期记忆是 `remember()`，用 LLM 从这一轮对话里把稳定事实抽出来、存盘。
- **读**：把相关的东西取出来、注入上下文。短期记忆是 `get_context()` 拼出 system 消息；长期记忆是 `recall()`，根据这一问把相关的几条事实找出来注入。

注意「写」和「读」中间隔着时间：**这一轮写进去的东西，是为了未来某一轮被读出来用**。所以写的时候要替未来的自己着想——存得太碎、太杂，将来就捞不准；存得有结构、够干净，将来才好检索。

写长期记忆时，把这三个问题想清楚，方向就不会错：

1. **存什么**——哪些算「稳定事实」，而不是一次性的闲聊？
2. **什么时候存**——每轮结尾抽取。
3. **取的时候怎么找到相关的那几条**——最简单是关键词命中，进阶可以换成向量召回。

下一节把这三个问题逐个掰开。

### 2.5 把长期记忆做对的几个难点 补

任务 2 的全部难度，几乎都集中在「写得准」和「读得准」这两头。

**① 什么才算「稳定事实」（写什么）**

判据就一个：**换了话题、换了窗口、过几天还成立吗？**

- 该记的：身份、长期偏好、约束条件——「我叫小明」「我是后端工程师」「我对花生过敏」「我习惯用中文回复」。
- 不该记的：一次性的、当下的、会过期的——「现在几点了」「帮我翻译这句话」「今天有点累」。这些塞进长期记忆只会变成噪音，把以后的召回越搞越脏。

实践里通常**让 LLM 来做抽取**：把这一轮对话喂给模型，让它判断「这里面有没有值得长期记住的事实，有就以结构化形式列出来」。这又回到了第 1 节的道理——你给它的抽取指令（prompt）写得清不清楚，直接决定它抽得准不准。

**② 抽出来的东西会重复、会打架（写得干净）**

- **去重**：用户可能在三个窗口里说过三次「我是 Python 工程师」，你不该存三条一模一样的。
- **冲突与更新**：人是会变的。上个月「我在用 Python」，这个月「我转去写 Go 了」——这俩不是两条并列事实，新的应该**覆盖/更新**旧的，否则召回时模型会拿到自相矛盾的信息。处理冲突是长期记忆里最容易被忽略、又最影响体验的一环。

**③ 怎么只捞出相关的那几条（读什么）**

长期记忆攒久了会有很多条，但每次注入只该带跟当前这一问相关的少数几条（受预算所限，见 2.2）。怎么判断「相关」，有两档做法：

- **关键词命中**（最简单，建议先用这个跑通）：当前问题里出现的词，去和存着的事实做匹配。够用，但死板——用户问「我的职业是什么」，而事实存的是「我是工程师」，字面对不上就可能漏掉。
- **向量召回 / 语义检索**（进阶）：把每条事实和当前问题都转成向量（embedding），按「语义接近程度」找最相关的几条。这样「职业」和「工程师」即使字面不同，也能凭意思接近被召回。这套「把内容向量化、按问题语义检索、再注入上下文」的思路，正是业界说的 **RAG（检索增强生成）**——长期记忆的「读」这一半，本质上就是一次小型 RAG。

**④ 别只会记，还得会忘（更新与隐私）**

一个能用的记忆系统，除了写和读，通常还要能**改**和**删**：事实过期了要能更新（见 ②），用户要求「忘掉我说过的某件事」时要能删。这既是体验问题，也是隐私和合规问题——你存的是真实用户的信息，让用户拿回控制权是底线。任务 2 不一定要求你把更新/删除做全，但心里要有这根弦。

> **任务 2 的「最小验证」之所以是「A 窗口说名字、B 窗口问名字」**，考的正是 2.3 表里那条：长期记忆按全局 `USER_ID` 共享，所以跨窗口还记得；要是你不小心按 `session_id` 存了，换个窗口立刻露馅。先用关键词命中把这条验证跑通，再考虑要不要上向量召回。
